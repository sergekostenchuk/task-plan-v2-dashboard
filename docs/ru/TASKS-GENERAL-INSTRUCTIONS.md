# TASKS GENERAL INSTRUCTIONS

Этот документ объясняет, как устроена новая система `TASK-PLAN v2`, зачем она нужна, как в ней работают задачи, агенты, ревью, тестирование, wiki-sync и HTML dashboard.

Документ написан для человека. Его цель: чтобы любой новый участник понял модель работы без чтения внутренних skill-файлов и без восстановления контекста из переписки.

Английская версия лежит здесь:

- [English version](../en/TASKS-GENERAL-INSTRUCTIONS.md)

## 1. Зачем вообще нужен TASK-PLAN v2

Обычный список задач с чекбоксами плохо подходит для AI-исполнения, потому что в нем часто не хватает:

- четкой цели фичи;
- границ scope;
- зависимостей;
- правил handoff между агентами;
- явного ревью и тестового gate;
- rollback-плана;
- правил записи знаний обратно в wiki;
- стабильной структуры для автоматической визуализации.

`TASK-PLAN v2` решает именно эту проблему.

Это не просто список задач, а **операционный документ управления разработкой**.

Он нужен, чтобы:

- готовить фичу до начала кодинга;
- раскладывать работу в нормализованные task blocks;
- запускать несколько агентов на одну задачу последовательно;
- не пропускать стадию review и test;
- не терять решения, риски и артефакты;
- автоматически строить поверх Markdown человекочитаемый dashboard.

## 2. Главная идея модели

Система состоит из трех уровней:

1. `FEATURE-PREPARATION.md`
2. `TASK-PLAN.md`
3. `TASK-DASHBOARD.html`

Их роли разные.

### `FEATURE-PREPARATION.md`

Это **pre-implementation layer**.

Здесь проверяется, готова ли фича к декомпозиции в задачи.

Если этот слой не заполнен, implementation-задачи не должны считаться `ready`.

### `TASK-PLAN.md`

Это **канонический source of truth**.

Именно этот файл:

- редактируется агентами;
- хранит feature-level и task-level структуру;
- фиксирует роли агентов, handoff’ы, approvals, tests, artifacts, rollback и wiki-sync.

Все реальные изменения сначала должны попадать сюда.

### `TASK-DASHBOARD.html`

Это **производный визуальный слой**.

Он нужен для человека, чтобы видеть:

- канбан по статусам;
- граф зависимостей;
- критический путь;
- текущего владельца задачи;
- какие review/test gates еще не пройдены;
- где лежат артефакты.

Важно:

- `HTML` не является источником истины;
- если `HTML` и `Markdown` расходятся, править нужно `Markdown`.

## 3. Как устроен TASK-PLAN v2

`TASK-PLAN v2` разделяет информацию по слоям.

### 3.1 Feature level

Это слой всей фичи или инициативы.

На этом уровне хранятся:

- `feature_id`
- `feature_title`
- `rationale`
- `priority`
- `status`
- `goal`
- `scope_in`
- `scope_out`
- `changed_subsystems`
- `constraints`
- `assumptions`
- `open_questions`
- `risks`
- `regression_risks`
- `security_privacy_notes`
- `non_functional_requirements`
- `milestones`
- `timebox`
- `wiki_pages_to_read_before`
- `wiki_pages_to_update_after`
- `wiki_facts_to_capture`
- `wiki_do_not_store`

Смысл этого уровня:

- понять, **зачем** вообще делается работа;
- держать общие рамки и риски;
- не дублировать одну и ту же вводную в каждой задаче.

### 3.2 Task level

Это слой конкретной задачи.

На этом уровне каждая задача получает нормализованный блок с полями:

- `task_id`
- `title`
- `rationale`
- `priority`
- `status`
- `dependencies`
- `blocked_by`
- `unblocks`
- `task_size`
- `decomposition_rule`
- `milestones`
- `timebox`
- `goal`
- `scope_in`
- `scope_out`
- `changed_subsystems`
- `candidate_files`
- `forbidden_areas`
- `constraints`
- `assumptions`
- `open_questions`
- `risks`
- `regression_risks`
- `security_privacy_notes`
- `non_functional_requirements`
- `acceptance_criteria`
- `acceptance_checks`
- `exit_criteria`
- `rollback_plan`

Смысл этого уровня:

- сделать задачу исполнимой;
- показать, где границы;
- показать, что считается завершением;
- не дать агенту "додумать" scope за пользователя.

### 3.3 Agent level

Это главное отличие новой модели от обычного task list.

Здесь фиксируется, **как одна и та же задача проходит через несколько агентов**.

Основные поля:

- `owner_role`
- `agent_sequence`
- `agent_contracts`
- `required_approvals`
- `max_review_loops`
- `escalation_rule`

### 3.4 Execution and reporting level

Этот слой отвечает за проверяемость результата.

Основные поля:

- `tests_required`
- `test_levels`
- `test_targets`
- `negative_tests`
- `fixtures`
- `test_data_origin`
- `oracle`
- `determinism_notes`
- `flakiness_risk`
- `commands_run`
- `stop_on_failure`
- `expected_artifacts`
- `code_artifacts`
- `test_artifacts`
- `review_artifacts`
- `artifact_locations`
- `observability`
- `decision_log`
- `summary_format`

Этот слой нужен, чтобы задача закрывалась не по ощущению, а по доказательствам.

## 4. Самое важное: одна задача и несколько агентов

По умолчанию система устроена **не как “главная задача + подтаски по ролям”**, а как:

**одна задача -> несколько агентов работают над ней последовательно**

Это значит:

- задача остается одной;
- `task_id` остается один;
- внутри нее меняется текущий владелец (`owner_role`);
- каждый следующий агент получает handoff от предыдущего.

Стандартная последовательность ролей:

1. `planner`
2. `implementer`
3. `reviewer`
4. `tester`
5. `docs_sync`

Если нужно, названия ролей можно адаптировать, например:

- `planner`
- `worker`
- `reviewer`
- `tester`
- `docs_sync`

Смысл последовательности:

- сначала уточнить и зафиксировать scope;
- затем сделать реализацию;
- затем отдельно проверить реализацию;
- затем отдельно прогнать тесты;
- затем синхронизировать знания и закрыть задачу.

## 5. Что именно означает “воркер -> ревьювер -> тестировщик”

Это не пожелание и не устная договоренность.

Это должно быть явно зафиксировано в задаче.

### `implementer` / `worker`

Его задача:

- сделать изменение в рамках scope;
- не выйти в forbidden areas;
- оставить code artifacts;
- передать дальше diff и implementation notes.

### `reviewer`

Он не пишет ту же задачу заново.

Он проверяет:

- корректность реализации;
- соблюдение scope;
- соблюдение boundary rules;
- соответствие acceptance criteria;
- не появились ли лишние изменения.

Он выдает один из двух исходов:

- `approve`
- `correction request`

То есть reviewer не просто “смотрит”, а является gate перед тестированием.

### `tester`

Тестировщик начинает только после review-этапа.

Он:

- исполняет нужные проверки;
- сохраняет test artifacts;
- фиксирует PASS/FAIL;
- не должен молча пропускать падения.

Именно поэтому в шаблоне есть:

- `test_levels`
- `test_targets`
- `commands_run`
- `stop_on_failure`
- `test_artifacts`

Идея простая:

- сначала код должен пройти review;
- потом он должен пройти test gate;
- только после этого задача может двигаться к закрытию.

## 6. Как это регулируется формально

Чтобы эта цепочка не была “на словах”, у задачи есть поля управления исполнением.

### `agent_sequence`

Определяет порядок ролей.

Например:

```md
agent_sequence:
- planner
- implementer
- reviewer
- tester
- docs_sync
```

### `owner_role`

Показывает, кто сейчас владеет задачей.

Это всегда одна роль.

Даже если над задачей в целом работают несколько агентов, в конкретный момент времени активный владелец один.

### `agent_contracts`

Это самый важный механизм handoff.

Для каждого агента отдельно фиксируется:

- когда он может начать работу;
- какие артефакты он получает на вход;
- что именно он должен сделать;
- что он должен выдать на выход;
- кому он передает результат дальше;
- на каком gate он останавливается;
- какие у него stop conditions.

Именно этот раздел превращает “несколько агентов” из идеи в управляемый pipeline.

### `required_approvals`

Это список обязательных approval gates.

Например:

- `design-review`
- `code-review`
- `qa-signoff`

Это не просто комментарии, а explicit checkpoints.

### `max_review_loops`

Показывает, сколько раз задача может ходить по кругу review -> correction -> review.

Это нужно, чтобы не зависнуть бесконечно.

### `escalation_rule`

Описывает, что происходит, если:

- ревью не проходит слишком много раз;
- вылез blocker;
- выяснилось, что задача была неправильно нарезана;
- изменился scope.

Пример логики:

- если review loops превысили лимит, эскалировать project owner или tech lead;
- если blocker ломает scope, вернуть задачу обратно planner.

## 7. Когда нужны отдельные задачи, а когда достаточно нескольких агентов внутри одной

Это критически важное правило.

### Оставляем одну задачу, если:

- цель единая;
- изменение атомарное;
- один pipeline `worker -> reviewer -> tester` логичен;
- не нужно независимо двигать части работы по статусам;
- dependency graph не требует отдельного task id.

### Разбиваем на несколько задач, если:

- работа реально распадается на разные deliverables;
- части можно делать независимо;
- у частей разные зависимости;
- у частей разные scope boundaries;
- одна часть разблокирует другую;
- одна часть слишком большая и требует отдельного контроля.

Иными словами:

- **роли внутри одной задачи** не равны подтаскам;
- **подтаски** появляются только тогда, когда это реально разные куски работы.

Роль `reviewer` почти никогда не должна оформляться как отдельная задача.

То же касается и `tester`.

Их правильное место обычно внутри `agent_sequence` одной задачи.

## 8. Статусы и их смысл

Рекомендуемый набор статусов:

- `draft`
- `ready`
- `in_progress`
- `blocked`
- `needs_review`
- `approved`
- `done`
- `dropped`

Как читать эти статусы:

### `draft`

Задача еще не готова к исполнению.

Обычно:

- не закрыт pre-implementation gate;
- не определен scope;
- есть open questions;
- не зафиксированы зависимости.

### `ready`

Задача подготовлена и может входить в execution pipeline.

### `in_progress`

Активная роль сейчас выполняет работу.

### `blocked`

Есть blocker, который мешает двигаться дальше.

### `needs_review`

Обычно означает, что `implementer` закончил свою часть и передал задачу на review.

### `approved`

Review gate пройден, задача может идти в тестирование или дальше.

### `done`

Задача может стать `done` только если:

- acceptance выполнены;
- обязательные approvals собраны;
- test gate пройден;
- артефакты сохранены;
- rollback понятен;
- wiki/docs-sync выполнен, если он required.

## 9. Как работает pre-implementation gate

До начала кодинга должна быть заполнена feature-preparation часть.

Она отвечает на вопросы:

- что это за фича;
- какую проблему она решает;
- кто пользователь;
- какие user intents поддерживаются;
- какой UI/UX поток ожидается;
- как выглядит preview;
- где rollback;
- какие subsystem затрагиваются;
- какие нужны API/contracts/events;
- какие acceptance;
- какие tests;
- какие regression risks;
- что входит в MVP;
- что откладывается;
- нужен ли feature flag;
- что писать в wiki.

Если этот слой пустой, то дальше обычно появляются проблемы:

- код пишется “в вакууме”;
- scope расползается;
- reviewers спорят о требованиях;
- тестирование не знает, что считать успехом.

Поэтому правило простое:

**сначала readiness, потом task decomposition**

## 10. Как устроены зависимости

У каждой задачи должны быть понятны:

- `dependencies`
- `blocked_by`
- `unblocks`

Разница между ними:

### `dependencies`

Это то, от чего задача формально зависит.

### `blocked_by`

Это активные текущие блокеры.

Сюда могут входить:

- другая задача;
- недостающий доступ;
- неготовый API;
- неразрешенный open question;
- отсутствие test data.

### `unblocks`

Это что именно станет возможно после завершения этой задачи.

Это поле особенно полезно для dashboard и критического пути.

## 11. Как работает test gate

В новой модели тестирование не является “желательной активностью”.

Оно описывается явно.

Минимальный набор:

- нужны ли тесты вообще;
- какие уровни тестов обязательны;
- что именно тестируем;
- какие negative tests нужны;
- откуда берутся fixtures;
- как определяется oracle успеха;
- какие команды реально запускались;
- что делать при падении.

Ключевой принцип:

**задача не считается завершенной только потому, что код написан**

Она должна пройти test gate.

Если `stop_on_failure: true`, то падение обязательной проверки блокирует закрытие задачи.

## 12. Как работает review gate

Review gate нужен для защиты от premature DONE.

Он предотвращает ситуацию:

- worker считает, что все сделал;
- а фактически поломал boundaries, acceptance, dependencies или quality bar.

Что должен проверять reviewer:

- задача решает именно нужную проблему;
- изменение не вышло за `scope_in`;
- `scope_out` не нарушен;
- `forbidden_areas` не затронуты;
- acceptance criteria действительно закрыты;
- rollback реалистичен;
- артефакты и handoff notes не фиктивны.

Если reviewer возвращает correction request, задача идет не в `done`, а обратно в implementation loop.

## 13. Как работает docs_sync и wiki_sync

После review и tests остается последний слой: синхронизация знаний.

Для этого есть поля:

- `wiki_pages_to_read_before`
- `wiki_pages_to_update_after`
- `wiki_facts_to_capture`
- `wiki_do_not_store`

Это нужно, чтобы:

- не терять решения;
- не забывать обновлять системные знания;
- не писать в wiki лишнее;
- не утекали секреты, токены, персональные данные и мусор.

Полезное правило:

- в wiki писать факты и решения;
- не писать туда шумные промежуточные рассуждения;
- не хранить секреты и личные данные.

## 14. Как человек должен работать с этой системой

Ниже практический сценарий.

### Шаг 1. Подготовить feature-preparation слой

Заполнить `FEATURE-PREPARATION.md`.

Нужно добиться, чтобы было понятно:

- что делаем;
- зачем;
- для кого;
- как выглядит user flow;
- что именно нужно проверить;
- что не входит в фичу.

### Шаг 2. Создать или обновить `TASK-PLAN.md`

После readiness разложить работу в задачи.

Каждая задача должна иметь:

- ясный `task_id`;
- явный `goal`;
- scope boundaries;
- dependencies;
- acceptance;
- rollback;
- agent pipeline.

### Шаг 3. Назначить последовательность агентов

Минимум:

- `implementer`
- `reviewer`
- `tester`

Чаще лучше:

- `planner`
- `implementer`
- `reviewer`
- `tester`
- `docs_sync`

### Шаг 4. Убедиться, что contracts прописаны явно

Без `agent_contracts` multi-agent модель быстро превращается обратно в хаос.

### Шаг 5. Вести задачу через статусы

Не перепрыгивать сразу в `done`.

Нормальная жизнь задачи выглядит так:

- `draft`
- `ready`
- `in_progress`
- `needs_review`
- `approved`
- `done`

или:

- `draft`
- `ready`
- `in_progress`
- `needs_review`
- `blocked`

или:

- `in_progress`
- `needs_review`
- `in_progress`
- `needs_review`
- `approved`

### Шаг 6. После изменения обновлять dashboard

Если есть `TASK-DASHBOARD.html`, он должен регенерироваться из Markdown.

Не наоборот.

## 15. Что считать хорошим TASK-PLAN v2

Хороший план:

- понятен человеку без внешнего контекста;
- не скрывает критические условия в prose;
- показывает, что делает каждый агент;
- явно разводит implementation, review и testing;
- имеет test и rollback логику;
- не позволяет закрыть задачу без доказательств;
- пригоден для генерации dashboard;
- сохраняет wiki-дисциплину.

Плохой план:

- состоит только из общих чекбоксов;
- не имеет feature prep;
- не содержит зависимостей;
- не различает worker и reviewer;
- не указывает, кто и когда должен тестировать;
- не определяет exit criteria;
- не фиксирует artifacts;
- не показывает, что считать success.

## 17. Короткая формула системы

Если объяснять совсем кратко:

- сначала готовим фичу;
- потом создаем канонический task plan;
- внутри одной задачи запускаем несколько агентов по очереди;
- worker не закрывает задачу сам;
- reviewer проверяет;
- tester подтверждает;
- docs/wiki sync фиксируют знания;
- HTML только визуализирует Markdown.

Это и есть новая рабочая модель.
